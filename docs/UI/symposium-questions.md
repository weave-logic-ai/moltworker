# Design Symposium: Expert Panel Questions

> Webview UI for Mentra Live Smart Glasses + OpenClaw AI Agent
> Generated by multi-panel design symposium simulation

---

## Panel 1: UX Researchers

Focus: User flows, cognitive load, context switching between audio and visual, attention management

### Q1.1 — Context Switching Cost
When the user receives an AI response via audio on the glasses and simultaneously sees updated content on the phone screen, what is the acceptable latency between the audio delivery and the visual update? If the phone lags behind the audio by more than 500ms, does the user perceive them as disconnected events rather than a unified response?
--- Not if there is signals showing it is coming, it may be that they cannot look at screen right away too.

### Q1.2 — Attention Split During Active Tasks
For the development agency use case, when a developer is reviewing code diffs on the phone while the agent is narrating changes through the glasses speaker, does the dual-channel delivery reduce comprehension compared to phone-only or audio-only? Should we offer a "focus mode" that suppresses one channel?
--- Always make it easy for a user to mute the agent and turn it into a text message.

### Q1.3 — Approval Fatigue
In workflows where the agent requests multiple sequential approvals (e.g., deploying a pipeline with 5 stages), at what point does approval fatigue set in? Should we offer batch approval ("approve all remaining") after a threshold, and what is the right threshold (3? 5?)?
--- When there is multiple approvals coming in sequence, use a survey type format for the user where the agent can help with the suggested answers and offering smart batched approval chunks.


### Q1.4 — Glance vs. Session Duration
The glasses design system targets 1-3 second glances. But the phone companion is for extended interaction. What is the expected session duration for each use case (dev, medical, field work, general)? This determines whether we optimize for quick-check or deep-engagement layouts.
--- Keep glaces and data collection minimal, smaller samples of more specific criteria to build a complete picture. 


### Q1.5 — Handoff Friction
When the glasses say "Details on phone," how quickly must the phone display the relevant content? Is there a maximum acceptable time from hearing the prompt to seeing the content? Research on watch-to-phone handoff suggests under 2 seconds, but the webview context may differ.
--- Handoff of workflows should be designed complete async. If there is something that the user was meant to look at, in this case if they did not look at it after a few moments, a task for the user should be created that can be followed up at any time.


### Q1.6 — Notification Hierarchy on Phone
When the agent pushes a notification while the user is actively reading a code diff on the phone, should the notification overlay the current view (interrupting), appear as a subtle banner (ambient), or queue silently until the user finishes? Does the answer change based on notification priority?
--- Overlay only in the case of an emergency or exception, alerts and silent queing is good. Usually user will be able to respond via voice and confirm if they want to have the context switch.


### Q1.7 — Conversation Thread Length
For the general AI assistant use case, at what message count does the conversation thread become unwieldy on a mobile screen? Should we auto-collapse older messages, paginate, or use an infinite scroll? What are the performance implications of each approach in a webview?
---Auto-Collapse as workflow progresses, try to collapse into logical units. Follow discords use of chats, including threads and infinite scroll.

### Q1.8 — Mental Model of the Agent
Users interact with OpenClaw via voice on glasses and visually on the phone. Do users perceive a single agent or two separate systems? If the mental model fractures, how do we reinforce continuity? Should the phone show real-time transcription of what the glasses are hearing/saying?
-- The local mentra linked agent should consider itself "in the glasses", or even as the glasses. Other agents can be brought in, they can use various voices(keep track so they persist) to achieve like you have a team.


### Q1.9 — Error Recovery Flows
When the WebSocket connection drops between phone and bridge, what is the user's expected recovery path? Do they expect automatic reconnection, or do they need a manual "reconnect" button? How do we communicate partial state (some data may be stale)?
--- automatic reconnection is fine, exposing the ability to reconnect is ok in the admin panel area. Use iconography to represent sync state, and queue backup etc. It should be clear to the user that the system is down. We want the user to be able to kick off a "doctor" type command so the whole thing can self-heal if it is not already.

### Q1.10 — One-Handed Operation Baseline
For field work and medical use cases, users often have only one hand free. What percentage of all webview interactions must be achievable one-handed? Is the answer 100%, or are there acceptable exceptions (e.g., initial setup, complex settings)?
---We can rate tasks based on how many hands they take 0 (voice only), 1 (phone screen), 2(laptop etc), they can even be negative hands because the user will not need to do anything, the ai will be saving time.

### Q1.11 — Learning Curve for New Users
How many interactions does a new user need before they understand the glasses-to-phone information flow? Should we include an onboarding tutorial, or is the interface self-explanatory enough? What is the dropout risk if onboarding takes more than 60 seconds?
--- No tutorials yet, just build it and use it. (its for me)


### Q1.12 — Multitasking Across Conversations
Can users maintain multiple active agent conversations (e.g., one for a DevOps task and one for a medical query)? If so, how do we prevent cross-contamination of context? Should conversation switching be explicit (tab-based) or implicit (agent auto-detects topic)?
---Client->Project->Workflow will wrap the context to keep that cross-contamination minimal to none. Switching to new workflows should be very clear to user, screen should always indicate what workflow<->project<->client they are on. It will be possible from some messages, emails, notifications to deep link into a workflow to answer a survey or review something etc.

## Panel 2: UI Designers

Focus: Component library, layout system, typography, color, information density, responsive design

### Q2.1 — Dark Theme Color Palette
The existing webview uses `#0f0f1a` (background), `#1a1a2e` (card), `#e94560` (accent), and `#4ecca3` (success). Should we maintain these exact values for continuity, or refine them for better WCAG contrast ratios? The current `#b0b0b0` on `#1a1a2e` yields approximately 5.2:1, which passes AA but not AAA.
Use the base theme for all colors https://github.com/shadcnstudio/shadcn-nextjs-track-landing-page-free this is perfectly designed.

### Q2.2 — Code Diff Rendering
Code diffs are a primary content type for the development use case. On a 375px-wide mobile screen, what is the maximum number of characters per line we can display in a monospace font at a readable size (minimum 12px)? Is side-by-side diff viable, or must we default to unified diff with inline additions/deletions?
--- https://github.com/AkshayMoolya/DiffMate use something like this to help with this. We may have to allow for different renderings, view full screen for instance. Inline editing/approval line by line is not needed on phone, if this has to happen user must use voice to tell agent, or get on laptop.


### Q2.3 — Information Density vs. Readability
The medical use case demands high information density (patient data, vitals, medication lists), while the field work use case demands large, scannable items (checklists with big tap targets). Can a single layout system serve both, or do we need use-case-specific density modes?
---Additional data and information services will be written as fast NN or vector embeddings as needed using ruvector. Small fast local models with good drift detection and specialized models etc, for instance this device: Cognitum Seed


### Q2.4 — Typography Scale
For a mobile webview embedded in a native app, should we use the system font stack (`-apple-system, BlinkMacSystemFont, system-ui`) for familiarity, or a custom typeface (e.g., Inter, IBM Plex) for brand consistency? What is the minimum body text size for extended reading in variable lighting conditions?
--- Geist and probably 16 point for text that must be read.

### Q2.5 — Card-Based vs. Full-Width Layout
The current prototype uses card-based layouts with rounded corners and borders. For content-heavy screens (conversation threads, task boards), do cards add unnecessary visual noise and reduce content area? Should we switch to full-width sections with subtle dividers for primary content views?
yes cleaner cards that look more like the flow design when needed.

### Q2.6 — Tab Bar Design
With four primary sections (Chat, Tasks, Files, Settings), a bottom tab bar is standard. But if we add more sections (Approvals, Logs, Media), should we use a scrollable tab bar, a hamburger menu, or a contextual tab system that shows different tabs based on the active use case?
---No hamburger or scrollable tab bar. Contextual tab system that is simple and fairly flat so switching around is not a big deal. (approvals under tasks, logs and media under files - not in the task menu, but on those pages)

### Q2.7 — Agent Activity Indicator Placement
The agent activity indicator (thinking, executing, waiting) needs to be visible regardless of which tab the user is on. Should it be integrated into the connection status bar (top), the tab bar (bottom), or as a floating overlay? Each has trade-offs for screen real estate and attention.
-- top, show iconographically with a deep dive into connection status' when clicking it.


### Q2.8 — Markdown Rendering Fidelity
The agent sends markdown responses that may include headers, lists, code blocks, tables, and images. What is the minimum viable markdown rendering we need for v1? Full GitHub-flavored markdown, or a subset (bold, italic, code, lists, links)?
-- full markdown, with mermaid and rich multimodal.


### Q2.9 — Responsive Breakpoints
The webview primarily targets phones (375px-428px), but could also be accessed on tablets or desktop browsers. Should we design exclusively for mobile-portrait, or include a tablet breakpoint (768px) where we could show side-by-side panels (conversation + tasks)?
---Mobile portrait and landscape, adding larger breakpoints for tablets and desktops/laptops is good but functionally keep it simple not crazy responsive design, SIMPLE!


### Q2.10 — Loading States and Skeleton Screens
When the agent is processing a request, should we show a skeleton screen (placeholder blocks), a spinner with status text ("Analyzing code..."), or a typing indicator (animated dots)? The choice affects perceived performance. Agent operations can take 2-30 seconds.
---skeleton blocks with spinners, text, animations, and details about how long it has been going as well as easy way to interupt and cancel (with confirm).


### Q2.11 — Empty States
What do Chat, Tasks, and Files look like when empty? Each needs a distinct empty state that guides the user toward the first action. Should empty states include example prompts, sample tasks, or instructional content?
---Chat, can basically be blank or show the abstract or mission statement for the current context. Tasks, prompt user to make some or celebrate that their are none to do, files should look like a file browser of the current project.


### Q2.12 — Visual Hierarchy for Approval Cards
Approval cards are high-stakes UI (deploying code, confirming medical decisions). How do we visually distinguish them from regular message cards? Color coding, border treatment, size, iconography, or a combination? How do we prevent accidental approval/rejection?
--- Modal pops after swipe left/right that outlines the decision, with buttons to change mind or follow up there in modal.offer a way to create a follow up item that cancels or follows up on the approval/disapproval.  Again make all of this easy, and clear to the user.

## Panel 3: Interaction Designers

Focus: Tactile feedback, connection status indicators, real-time state, gesture/swipe patterns, notification design

### Q3.1 — Swipe-to-Approve Gesture
For approval cards, is swipe-right-to-approve / swipe-left-to-reject intuitive enough, or does it risk accidental actions? Should we require a swipe + confirmation tap (two-step), or is the swipe threshold distance sufficient protection? What is the right threshold (30% of card width? 50%)? 
--- At 50% pop the decision approval modal

### Q3.2 — Connection Status Pulse
The connection status bar should show a "pulse" animation when connected. What is the right pulse frequency? Too fast (1Hz) feels anxious; too slow (0.1Hz) feels dead. Medical device UIs use 1-2 second cycles. Should the pulse stop when the user is actively interacting (to avoid distraction)?
-- Use some sort of qos metric to give a # of bars, not a pulse. Same with other connected devices if there are any, show them via iconography like bluetooth, network, battery, etc. Speaking of Battery, we want to show this for the glasses in that status bar.

### Q3.3 — WebSocket Reconnection UX
When the WebSocket drops, how quickly should the UI reflect the disconnection? Immediately (jarring), after a 2-second grace period (smooth but possibly stale), or only after a reconnection attempt fails? What visual treatment differentiates "reconnecting..." from "disconnected"?
-- a reasonable delay based on how much buffering can be done if the websocket drops. A grace period is certainly needed to stop jarring when the websocket is flapping.


### Q3.4 — Pull-to-Refresh Behavior
Should the conversation view support pull-to-refresh to fetch missed messages, or should the WebSocket guarantee delivery? If the user pulls to refresh, should it re-fetch the entire conversation or only messages since the last known timestamp?
---Entire conversation tree, some branches can be truncated for "closed sections", but they do NOT need to be.


### Q3.5 — Haptic Feedback in Webview
Native haptic feedback is not available in webviews on all platforms. Should we implement visual-only feedback for taps and swipes, or attempt to use the Vibration API where available? If haptics are unreliable, how do we compensate with visual and audio cues?
---haptic should always supliment visual and audio when possible


### Q3.6 — Long-Press Interactions
Are there any interactions that warrant long-press (e.g., long-press a message to copy, long-press a task to reorder)? Long-press is discoverable only through habit. Should we provide alternative paths (context menu dots, explicit buttons) for all long-press actions?
---no long press yet. keep it simple


### Q3.7 — Scroll Behavior in Conversation
When a new message arrives while the user is scrolled up reading history, should the view auto-scroll to the bottom (interrupting reading) or show a "New message" pill that the user can tap to jump down? What if the new message is an approval request (high priority)?
--- newmessage pill, high priority can be a more noticible pill and other notification.

### Q3.8 — Transition Animations
When switching between tabs (Chat, Tasks, Files, Settings), should we use slide transitions (directional), fade transitions (neutral), or instant switches (fastest)? The choice affects perceived speed and spatial orientation. What duration is optimal (150ms? 250ms? 300ms)?
--- fastest, keep animations minimal except for functional animations in visualizations or UX.

### Q3.9 — Real-Time Transcription Display
If we show real-time transcription of what the glasses microphone is picking up, how do we display interim (non-final) transcription results? Greyed-out text that solidifies? A separate "listening" area? This text changes rapidly and could be visually noisy.
Where the user would normally type, which they still can, use greyed out text that solidifies. Rapid changes here is ok.

### Q3.10 — Notification Toast Positioning
Notification toasts from the agent (task completed, error occurred, approval needed) should appear where? Top of screen (standard iOS), bottom of screen (above tab bar, closer to thumb), or center (maximum attention)? Should they auto-dismiss, and after how many seconds?
bottom over bottom bar, make them easy to close and able to be clicked on to get more details. Auto dismiss after 2 seconds, they will still be in notications, and it can animate to make it look like it is dropping into there.


### Q3.11 — Gesture Conflicts with Native App
Since the webview runs inside the MentraOS native app, do standard webview gestures (swipe from edge to go back, pinch to zoom) conflict with native app gestures? Should we disable pinch-to-zoom entirely and handle edge swipes carefully to avoid triggering native navigation?
no pinch to zoom, instead use a hold and make a magnification glass above finger. Edges left and right are not used in an app, leave top and bottom alone


### Q3.12 — Offline Queue Visualization
When the device goes offline, queued actions (pending approvals, unsent messages) need visual treatment. Should we show them inline with a "pending" badge, in a separate "outbox" view, or grayed out with a sync icon? How do we handle conflicts when the queue syncs?
--- Don't allow actions if the gateway is down, this way there is no pending queue to sync.

## Panel 4: OpenClaw/AI Agent Experts

Focus: Content types the agent pushes, conversation threading, tool output display

### Q4.1 — Content Type Taxonomy
The agent can produce: plain text, markdown, code blocks, diffs, images, video frames, task lists, approval requests, error logs, deployment status, test results, and structured data (JSON/tables). Should each content type have a dedicated renderer, or should we use a universal markdown renderer with embedded custom blocks?
--- https://elements.ai-sdk.dev/components/ Use these components to help manage all of this.

### Q4.2 — Tool Output Display
When the agent executes a tool (e.g., runs tests, deploys code, queries a database), the output can be verbose (hundreds of lines). Should we show the full output inline, collapse it behind an expandable section, or show a summary with a "view full output" link? What is the right default for each tool type?
--- https://elements.ai-sdk.dev/components/agent handles all of that.


### Q4.3 — Streaming vs. Complete Responses
Should the webview display agent responses as they stream (token by token, like ChatGPT), or wait for the complete response? Streaming feels faster but complicates layout (content height changes). For code diffs specifically, streaming partial diffs is likely confusing.
 --- https://elements.ai-sdk.dev/components/chain-of-thought

### Q4.4 — Conversation Threading Model
When the agent works on multiple tasks simultaneously (e.g., running tests while answering a question), should responses be interleaved in a single thread (chronological), separated into sub-threads (per task), or shown in separate tabs? How does this interact with the glasses audio, which is necessarily single-threaded?
--- Tasks should have a task component: https://elements.ai-sdk.dev/components/task additionally use https://elements.ai-sdk.dev/components/queue to queue up multiple tasks. Keep the rest interleaved into single thread sliding queue and task components to current chronological spot until they finish. 

### Q4.5 — Approval Request Schema
What metadata must an approval card contain? At minimum: title, description, risk level, affected resources, diff preview, agent confidence, timestamp, expiration. Should approvals expire after a timeout, and if so, what is the default (5 minutes? 1 hour? never)?
--- Sometimes it may be a list of https://elements.ai-sdk.dev/components/suggestion or a quick decision: https://elements.ai-sdk.dev/components/confirmation it could be a deeper appoval card with details, a form perhaps the user fills out and then approves is possible. Approvals can't expire, but they can go stale and be removed by the agent or by the user. All of this should be clear to user.

### Q4.6 — Agent State Machine Visibility
The agent has states: idle, thinking, executing tool, waiting for approval, error. Should the user see all state transitions, or only meaningful ones? If the agent rapidly cycles through think-execute-think-execute, do we debounce the status indicator to avoid flickering?
---Use https://elements.ai-sdk.dev/components/persona and https://elements.ai-sdk.dev/components/chain-of-thought and debounce is ok to stop rapid cycling and flickering.


### Q4.7 — Image and Video from Glasses Camera
When the user takes a photo via the glasses and the agent analyzes it, should the webview show the photo alongside the analysis? For video (streaming camera), should the webview show a live preview, or only show captured frames when the agent references them?
--- Show the photo(s) when analyzing. Video and camera when being used should have controls on screen, user should not have a preview.

### Q4.8 — Context Window Visibility
Should the user see how much of the agent's context window is consumed? This is relevant for developers who understand token limits. A progress bar showing "Context: 42% used" could help power users decide when to start a new conversation.
 - https://elements.ai-sdk.dev/components/context should be used.

### Q4.9 — Agent Configuration from Webview
Should the webview allow users to configure agent behavior (system prompt, model selection, temperature, tool permissions)? If so, should these settings be per-conversation or global? How do we prevent users from accidentally breaking the agent by changing advanced settings?
---Yes they should. Keep deltas of all changes on settings/agents/etc, so rollback of settings is possible.

### Q4.10 — Error Presentation
When the agent encounters an error (API timeout, tool failure, permission denied), how much detail should the user see? Developers want stack traces; medical professionals want a simple "try again" button. Should error verbosity be configurable per use case?
---Try again is good, a ? icon or magnifying glass can pop logs in a chrome window or something as well. Error verbosity must be able to be adjusted.


### Q4.11 — Multi-Modal Input from Phone
Beyond voice on glasses, can the user send text, images, or files from the phone webview to the agent? If so, how does this interact with the glasses audio channel? Does the agent acknowledge phone input via the glasses speaker?
--- Use attachments https://elements.ai-sdk.dev/components/attachments https://elements.ai-sdk.dev/components/prompt-input should be used to wrap the input area, and there should be controls for muting, muting mic, stopping playback etc exposed in the prompt input area.
 
### Q4.12 — Conversation History Persistence
How far back should conversation history be available? Last session only, last 24 hours, last 30 days, or indefinite? Where is history stored (device, server, both)? Can conversations be exported, searched, or shared?
--- Conversations should be indefinetly stored, and we will use a channel in openclaw to handle this.

## Panel 5: Accessibility and Cross-Platform

Focus: WCAG compliance, one-handed operation, varying light conditions, screen sizes

### Q5.1 — WCAG AA vs. AAA Target
For a professional tool used in medical and field work contexts, should we target WCAG 2.1 AA (minimum 4.5:1 contrast for normal text) or AAA (7:1)? Given that field workers may use the phone in direct sunlight, AAA may be necessary for outdoor legibility.
--- AA is fine, we can add a high contrast theme (like dark mode if need be)

### Q5.2 — Dynamic Text Sizing
Should the webview respect the device's system text size settings (Dynamic Type on iOS, font scale on Android)? If so, how do we prevent layout breakage at 200% text size? Code diffs and monospace content are particularly vulnerable to text scaling.

### Q5.3 — One-Handed Reach Zones
On a standard phone (375px wide), the bottom 40% of the screen is the comfortable one-handed reach zone. Are all primary actions (approve/reject, send message, switch tabs) within this zone? Should we offer a left-hand mode that mirrors the interface?
--yes, no left hand for now.

### Q5.4 — Screen Reader Compatibility
Webviews have inconsistent screen reader support across platforms. Should we add explicit ARIA roles and labels to all interactive elements? For the conversation thread, how do we label AI vs. user messages for VoiceOver/TalkBack? Should code diffs be announced line-by-line?
-- nope, not needed. ARIA screenreader support is not critical on this project.

### Q5.5 — Color Blindness Considerations
The current palette uses red (`#e94560`) for accent and green (`#4ecca3`) for success. This red-green pairing is problematic for deuteranopia (8% of males). Should we add secondary indicators (icons, patterns, position) so that no information is conveyed by color alone?
--- pallet should follow example theme

### Q5.6 — Varying Light Conditions
Field workers and medical professionals use phones in environments ranging from pitch-dark operating rooms to noon sunlight. Should we support an auto-brightness-aware theme that increases contrast in bright conditions? Or should we stick to a single high-contrast dark theme?
--- dark/light theme initially with a possible high contrast version later


### Q5.7 — Touch Target Sizing
WCAG 2.5.5 requires 44x44px minimum touch targets. The approval card swipe area, tab bar buttons, and action buttons must all meet this. Are there any components in the current design that fall below this threshold?
--- no but this shows how big status icons etc need to be,

### Q5.8 — Reduced Motion Support
Users with vestibular disorders may be affected by animations (pulse indicators, slide transitions, swipe feedback). Should we detect `prefers-reduced-motion` and provide static alternatives for all animations? What does the connection pulse look like without animation?
--- we won't use too many of these in the first place.

### Q5.9 — Offline Accessibility
When the device loses connectivity, is all previously loaded content still accessible and navigable? Can the user still browse conversation history, view completed tasks, and read cached diffs? What content requires connectivity to display?
--- without connection to gateway it will not work. Glasses can lose connection without breaking that.


### Q5.10 — Internationalization Readiness
Will the webview need to support RTL languages (Arabic, Hebrew) or CJK scripts (Chinese, Japanese, Korean)? If so, how does this affect the swipe-to-approve pattern (swipe direction semantics are culturally loaded) and the monospace code diff rendering?
 --- no

### Q5.11 — Focus Management
When a notification toast appears and auto-dismisses, where does keyboard/screen-reader focus go? If an approval card appears while the user is mid-scroll, should focus be forcibly moved to the approval (disruptive but ensures awareness) or left in place (respectful but may be missed)?
--- never steal focus unless user chooses it with an action. In action does not count. Notify and escalate through a repeating message later if the user did not do the action and it still needs to be done.

### Q5.12 — Cross-Platform WebView Differences
iOS WKWebView and Android WebView have different CSS rendering, JavaScript API availability, and gesture handling. What is our testing matrix? Do we test on: Safari iOS 16+, Chrome Android 12+, and Samsung Internet? How do we handle features unavailable in older webviews (e.g., CSS `backdrop-filter`)?
--- chromium browser inside the MentraOS app is the output.

## Cross-Panel Priority Questions

These questions span multiple panels and require coordinated answers:

1. **Primary use case for v1**: Which use case (dev, medical, field, general) do we optimize for first? The answer drives every design decision.
--- dev (AI development Agency)

2. **Real-time vs. polling**: Do we commit to WebSocket for all real-time updates, or provide a polling fallback? This affects interaction patterns, loading states, and offline behavior.
Websockets where real-time is needed and around the session handshaking. docs/mentraOS/message-types.md shows all the different types of inteactions, and docs/mentraOS/cloud-overview.md explains the networking.

3. **Authentication flow**: The webview uses `@mentra/react` with `MentraAuthProvider`. How does this affect the first-load experience? Is there a loading state before auth completes?
Small loading state with only the logo photo showing.

4. **Performance budget**: What is the maximum acceptable time-to-interactive for the webview? 1 second? 3 seconds? This constrains the complexity of the initial render.
--- User actions need to fast to see interactions. Slow load is fine, as long as it is only intiatal. 

5. **Native bridge API**: Does the MentraOS native app expose any bridge APIs to the webview (haptics, notifications, deep links)? This determines which interaction patterns are possible.
--- you can use Android Smart Glasses Client library for this: https://docs.mentraglass.com/os-devs/asg-client/README#overview
